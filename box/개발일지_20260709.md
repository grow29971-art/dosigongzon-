## 1. 보안 감사 — "1시간동안 해킹해봐" 요청 대응

57개 API 라우트 전수 점검. 발견·수정 순으로.

### ① 수동 배틀 결과 위조 (심각도 높음, 커밋 `0da4ed6`)

**진단**: AUTO 모드 배틀은 서버가 직접 시뮬레이션해서 안전한데, MANUAL 모드는 클라이언트가 턴을 진행하고 `winner`·`opp_cat_id`만 `/record`에 보고 — 서버가 이 값을 그대로 신뢰했음. 매칭 API를 거치지 않고 `/record`를 직접 반복 호출하면 코인·경험치 무한 파밍이 가능했고, 더 심각하게는 **임의의 `opp_cat_id`(다른 유저의 실제 카드 UUID)를 넣어 그 유저 카드의 `win_streak`을 조작**할 수 있었음.

**적용**: `lib/battle-token.ts` — HMAC-SHA256 서명 토큰. 매칭 API(`/api/cats/card-battle`)가 상대를 정할 때 토큰을 발급해서 클라이언트에 내려주고, `/record`가 토큰을 검증해야만 보상 지급. 서명 키는 이미 서버에만 있는 `SUPABASE_SERVICE_ROLE_KEY` 재사용. `is_boss`도 토큰에 서명된 값만 신뢰(클라이언트가 보낸 값 무시).

**한계**: "실제로 매칭된 상대에 대해서만" 기록을 인정하게 막았을 뿐, "정말로 이겼는지"까지는 못 막음 — 완전한 서버측 배틀 시뮬레이션 재작성 없이는 자기 승리 자체는 조작 가능. 후속 과제로 남김.

### ② 나머지 5건 (커밋 `57713ce`, `2f005e4`)

- `/api/cats/check-rarity-upgrade` — 인증 체크 자체가 없었음(프로젝트 규칙 위반). 로그인 필수로 추가.
- `/api/coins/care-bonus` — 실제 돌봄일지 작성 여부 확인 없이 POST만 오면 지급 → 반복 호출로 하루 최대 10코인 공짜 파밍 가능했음. 오늘 실제로 쓴 돌봄일지 개수(KST 기준)를 서버에서 직접 세어 그 범위 안에서만 지급하도록 수정.
- `/api/shop/use-item` — 재고 차감이 원자적이지 않아 동시 요청 시 경쟁 상태. `gt("quantity",0)` 필터로 최소 방어.
- `/api/chat` — AI 챗봇 분당 대화 한도(Lv1-2=10, Lv3-4=20, Lv5+=30)를 요청 body의 `level`을 그대로 신뢰 → `{level:999}`만 보내면 아무 계정이나 최고 등급 받아감(Gemini API 비용 문제). 서버에서 카드 등록 수 기반 하한 점수를 직접 조회하도록 변경.
- `cats` 테이블 RLS는 `caretaker_id = auth.uid()`로 제대로 막혀있음을 확인(가장 근본적인 방어선 검증) — 관리자 라우트 전부 `admins` 테이블 조회로 권한 확인 중임도 확인.

## 2. 성능 3건

### ① 카드창고 페이지 순차 쿼리 → 병렬 (커밋 `af398ff`)

**진단**: 도감/장착슬롯/보유아이템 기능을 세션 내내 하나씩 추가하면서 매번 "기존 로직 안 건드리게" try/catch로 따로 await를 붙였는데, 결과적으로 3개 쿼리가 순차 실행되고 있었음. "앱이 느려졌다"는 사용자 체감의 실제 원인.

**적용**: `Promise.allSettled`로 병렬화. 마이그레이션 전이라 컬럼 없는 항목이 실패해도 나머지는 그대로 반영되는 동작은 유지.

### ② 홈페이지 SSR 블로킹 → Suspense 스트리밍 (커밋 `7c89aeb`)

**진단**: 실측(`curl` 타이밍)으로 홈이 다른 페이지보다 확실히 느림을 확인 — 홈 1.5~3초 vs 마이페이지/지도/커뮤니티/가이드 0.4~1초. 로그인 확인 + HOT게시글·입양섹션·1000명이벤트 3개 섹션이 전부 끝나야 응답이 나가던 구조.

**적용**: 3개 섹션을 각각 `<Suspense fallback={null}>`로 감싸서 로그인 확인이 끝나는 대로 셸을 먼저 스트리밍하고 나머지는 준비되는 대로 채워지도록 변경.

### ③ 장착/해제 API — DB 함수로 왕복 1번 (커밋 `47e336f`)

**진단**: 이미 배치 조회·병렬 쓰기까지 최적화해놔도 "로그인 확인 → 카드 소유권 확인 → 재고 조회 → 갱신" 최소 3~4번의 순차 API↔DB 왕복이 남아있었음.

**적용**: `box/supabase_equip_item_rpc_migration.sql`의 `equip_item_atomic()` Postgres 함수 하나로 소유권 확인부터 재고 증감까지 트랜잭션 안에서 처리. RPC 미배포 시(마이그레이션 전) 예전 다단계 방식으로 자동 폴백. 배포 후 실측: 장착 83ms, 해제 53ms.

**참고**: 사용자가 SQL Editor에서 실행할 때 편집기에 이전 시도 내용이 안 지워진 채 겹쳐 붙여져서 "syntax error near p_cat_id" → "INTO specified more than once" 순으로 다른 에러가 남. 편집기를 완전히 비우고 재시도하니 해결.

## 3. 장비창 UI 재설계 (2단계)

**1차 시도** (커밋 `2f1e64d`): 오각형으로 사진 주위에 아이콘을 흩뿌려놓던 원래 배치가 "구성이 촌스럽다"는 피드백 → 사진을 히어로로 분리하고 3행 2열 컬러 카드 그리드로 변경. 부위별 테마색(급소=보라, 공격=주황, 생명=핑크, 방어=블루, 회피=그린, 테두리=골드) 도입.

**2차 시도** (커밋 `b06ecc9`): "더 별로임, 디아블로처럼 안 되나" 피드백 → 진짜 디아블로 장비창처럼 재설계. `clip-path`로 8각형(각진) 소켓을 만들고, 초상화를 중심에 둔 육각 배치(부위 5개 + 테두리)로 되돌리되, 이번엔 이중 테두리 + 베벨 그림자(inset highlight/shadow)로 "보석이 박힌 소켓" 느낌을 냄. 초상화는 보라·블루·골드가 도는 `conic-gradient` 링 프레임.

카드창고 상세 모달의 부위별 장비창 리스트도 같은 배색으로 통일(커밋 `2f1e64d`에 포함).
