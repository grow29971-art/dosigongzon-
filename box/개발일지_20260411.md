## 1. OAuth 이슈 디버깅 (10:00 ~ 11:00)

### 증상
- 친구가 카카오톡으로 보낸 링크 눌러서 로그인 시도 → 구글은 `disallowed_useragent (403)`, 카카오는 `KOE205 잘못된 요청`

### 원인 분석
1. **Google OAuth**: 2021년 정책으로 모든 임베디드 웹뷰에서 OAuth 차단. 카톡/페북/인스타 웹뷰 모두 해당. Google 측 정책이라 앱 개발자가 우회 불가능.
2. **Kakao KOE205**: 카카오 개발자 콘솔의 `Default Rest API Key` 내부에 "카카오 로그인 리다이렉트 URI" 필드가 있는데, `https://sozxbnvgsougkliibnxl.supabase.co/auth/v1/callback` 이 미등록 상태였음. 어제는 카카오맵 제품만 활성화했고 Login Redirect URI는 별도였음.

### 해결
- **카카오 Redirect URI 등록**: REST API 키 편집 화면 > 카카오 로그인 리다이렉트 URI에 Supabase 콜백 URL 추가.
- **인앱 브라우저 감지 + 외부 브라우저 탈출 (신규 코드)**
  - `lib/in-app-browser.ts`: UA 문자열에서 카톡/페북/인스타/네이버/라인/다음/밴드 8종 감지
  - `detectOS()`: android/ios/other
  - `openInExternalBrowser()`: 
    - Android → `intent://city-amber-omega.vercel.app#Intent;scheme=https;package=com.android.chrome;end` 로 크롬 강제 오픈
    - iOS 카카오톡 → `kakaotalk://web/openExternal?url=...` 전용 스킴
    - iOS 기타 → URL 클립보드 복사 + 수동 안내
  - `app/login/page.tsx`: 페이지 로드 시 `detectInAppBrowser()` 호출, 감지되면 로고 위에 빨간 배너 표시 (`[카카오톡] 안에서는 소셜 로그인이 안 돼요 — 크롬/사파리에서 열기` 버튼)

### 배움
- 크롬 번역 기능이 **Supabase SQL Editor의 monaco 에디터까지 한글로 번역**해서 SQL 키워드가 "여기서", "삽입합니다" 로 바뀌는 버그가 여러 번 발목을 잡음. 사용자님이 번역 모드로 열어둔 줄 모르고 "Chrome 번역 OFF" 주석을 SQL 파일 맨 위에 박기 시작함.

## 오늘의 배움

1. **OAuth는 네트워크 환경이 전부**. 브라우저 UA, 리다이렉트 URI, 허용 도메인, Provider 설정, 앱 상태(개발/배포), 번역 기능의 텍스트 변조까지. 코드가 맞아도 환경이 하나라도 틀어지면 전부 막힘. 에러 메시지를 무조건 신뢰하지 말고 **네트워크 탭 → URL → Status → Response body** 순으로 층층이 확인해야 함.

2. **Chrome 자동 번역이 개발자 도구를 망친다**. Monaco/CodeMirror 에디터도 번역 대상이라서 SQL/JS 키워드가 전부 한글로 바뀜. 사용자님께 `supabase.com` 을 번역 예외 사이트로 등록하도록 안내하는 게 반복 실수 방지에 필요. 모든 SQL 마이그레이션 파일 헤더에 `⚠ Chrome 번역 OFF` 주석 박기 시작함.

3. **RLS + PL/pgSQL 함수가 운영 MVP에 가장 비용 대비 효과적**. 유저 정지, 투표 카운트, 관리자 권한 체크 등을 전부 DB 레벨에서 해결하니 클라이언트 위조 걱정 없음. 특히 `is_user_not_suspended` 처럼 재사용 가능한 helper function은 여러 insert 정책에서 공유 가능.

4. **스냅샷 패턴이 소셜 앱에 기본**. 댓글의 author_name, author_avatar_url, author_level 을 외래 조인 대신 insert 시점에 snapshot해서 저장. 프로필 변경 시 과거 댓글 반영 안 되는 단점이 있지만, 조회 쿼리 단순화 + 삭제된 유저 정보 보존 등 장점이 훨씬 큼. 트위터/인스타 표준 패턴.

5. **관리자 권한 부트스트랩 문제**. admins 테이블에 insert 정책을 막으면 처음 admin 추가를 클라이언트에서 못함 → service_role(SQL Editor)로만 가능하도록 설계. 이 보안-편의 트레이드오프가 합당. 단, 마이그레이션 파일에 이메일 하드코딩해서 첫 admin 자동 등록.

6. **낙관적 업데이트 + 롤백이 소셜 인터랙션 핵심**. 좋아요 누를 때마다 서버 응답 기다리면 UX 못 씀. 즉시 UI 반영 + 실패 시 `catch` 블록에서 상태 되돌림. 투표 전환(like → dislike) 케이스는 prev 상태 +/- 처리가 복잡해서 테스트 중요.

7. **집단 검증이 악용 방지에 효과적**. 학대 경보를 "1건이라도 있으면 뜨게" 만들면 장난/오해로 경보가 남발됨. "2건 이상 + 30일 �window" 로 임계치 설정하니 자연스러운 합의 검증 구조가 생김. 운영 규칙이 그 자체로 디자인이 됨.

8. **디자인 언어 통일이 뒤늦게 효과를 냄**. 보호지침 먼저 vibrant 8색 + 화이트 카드 + glow로 했더니, 커뮤니티/마이페이지/병원/홈 전부 같은 패턴으로 재사용. 새 페이지 만들 때마다 "카드 컨테이너 스타일 찾기" 고민이 사라짐. 일관성이 속도를 낳음.

9. **"성우" 하드코딩 사건**. AI 집사 API에 하드코딩된 이름이 10군데 넘게 있었음. 과거에 내가 직접 쓴 코드를 지금 와서 본인 계정으로 테스트하니까 보였고, 다른 사용자가 쓰면 "저 사람 이름 왜 저래요" 가 될 뻔했음. **플레이스홀더 + 런타임 치환** 패턴을 처음부터 고수해야 함. 앞으로 이름/지역/개인정보 넣을 일 있으면 `{name}` 부터 쓸 것.

10. **어제 6시간 + 오늘 8시간 = 하루 평균 7시간**. 피처 20+ 개, SQL 마이그레이션 11개, 배포 7번. 비교적 큰 속도지만 이게 가능한 이유는 **디자인 언어 고정 + Supabase RLS 패턴 재사용 + `lib/*-repo.ts` 파일 구조 표준화** 덕분. 구조를 한 번 정하면 같은 패턴을 여러 번 굴릴 수 있음.
