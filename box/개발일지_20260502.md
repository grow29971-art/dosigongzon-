## 1. 길고양이 뉴스 자동 크롤러

문제: admin이 직접 큐레이션하는 `news` 테이블만 있어서 첫 화면이 정적. 매일 새 콘텐츠를 직접 등록하는 건 부담.

설계 결정:
- **저작권 안전을 위해 본문 저장 X.** Google News RSS의 제목·요약 2줄·썸네일·원문 링크만 카드 형태로 저장. 클릭 시 원문으로 이동.
- 별도 테이블/페이지 안 만들고 **기존 `news` 테이블 재활용** — `external_url`, `image_url`, `badge_type`이 이미 있어서 거의 그대로 맞아떨어짐.
- `source_url`을 UNIQUE로 두어 중복 자동 무시.
- **30일 지난 자동수집 글은 매 cron 실행마다 자동 정리** (`pinned=true`는 보존). admin이 수동으로 쓴 글은 영원히 살아있음.

작업:
- `box/supabase_news_crawler_migration.sql` — `news`에 `source_url(UNIQUE)`, `source_name`, `auto_imported` 추가
- `lib/news-crawler.ts` — 외부 의존성 없이 정규식으로 RSS XML 파싱. 키워드 7종(`길고양이`, `길냥이`, `캣맘`, `TNR 길고양이`, `유기묘 구조`, `고양이 학대`, `동물보호법 고양이`) 순차 수집 후 dedupe.
- 화이트리스트(`길고양이/캣맘/TNR/...`) + 블랙리스트(`분양/판매/광고/...`) 필터로 노이즈 컷.
- badge 자동 분류: `구조/학대/사고` → `urgent`, `TNR/중성화` → `tnr`, `법/조례` → `law`, `축제/캠페인` → `event`, 그 외 → `notice`.
- `app/api/cron/news-crawl/route.ts` — 매일 KST 09:00 (UTC 00:00) 실행. CRON_SECRET 검증, service_role로 INSERT.
- `app/(main)/admin/news/page.tsx` — 자동수집 항목에 "자동수집" 보라색 뱃지 + 출처명 표시.

기술 메모: Google News RSS의 `link`는 자기네 redirect URL이라 같은 기사라도 매번 다른 URL이 나올 수 있음. 일단 그대로 저장해서 dedupe하고, 노이즈 보고 필요하면 redirect 따라가는 식으로 강화 예정.

## 2. 관리자 무한 좋아요 누적 모드

요청: "관리자 계정은 좋아요 중복으로 찍을 수 있게."

설계: RPC(`post_vote_update`) 자체는 +1만 받게 두고, 클라이언트에서 admin이면 토글 없이 매번 `(deltaLike: 1)` 호출. 한 번 클릭 = +1, 100번 클릭 = +100. UI 시각 피드백은 일부러 비활성 상태로 두어 "무한히 누를 수 있다"는 신호.

- DB / RPC 변경 없음.

영향: 콘텐츠가 적은 초기에 운영자가 "조명" 줄 게시글에 인위적 부스트 가능. 정식 사용자에겐 영향 없음.

## 3. 게시글 조회수 baseline 부스트

데이터 패치 SQL. dummy seed로 만든 글들이 조회수 0~몇 정도라 사이트가 정적으로 보임.

```sql
UPDATE public.posts
   SET view_count = GREATEST(view_count, 50 + floor(random() * 51)::int);
```

GREATEST 덕분에 이미 100 넘는 인기글은 그대로 유지. 50 미만인 글만 50~100 사이 랜덤값으로 끌어올림. 일회성.

파일: `box/supabase_posts_view_count_boost.sql`

영향: 본인이 쓴 글에 반응이 오면 즉시 알림 → 재방문 강력한 트리거 1순위.

## 5. 좋아요 묶음 알림 (재방문 트리거 #2)

설계 고민: 좋아요는 가벼운 신호. 즉시 푸시는 시끄러움. 하루 1회 묶음 발송이 적절.

문제: posts 테이블엔 like_count(누적 카운터)만 있고 "누가 언제 눌렀는지" 기록 없음. → 스냅샷 패턴으로 해결.

작업:
- `box/supabase_posts_like_snapshot_migration.sql` — `like_count_snapshot int`, `like_snapshot_at timestamptz` 추가. 첫 도입 시 현재 like_count로 초기화.
- `app/api/cron/like-digest/route.ts` — 매일 KST 12:00 (UTC 03:00). 모든 글에서 `like_count > like_count_snapshot`인 글을 작성자별로 그룹. 한 글이면 `'"제목" 글에 새 좋아요 N개'`, 여러 글이면 `'내 글 M개에 새 좋아요 N개'`. 발송 후 스냅샷 갱신.
- engagement-push 패턴 그대로 — webpush 직접 호출, 410/404면 구독 자동 정리.

기술 메모: PostgREST는 column-to-column 비교(`>` 연산자에 다른 컬럼) 지원 안 함 → JS에서 in-memory 필터. 글 수 적어서(수천) 문제 없음. 수만 단위 되면 RPC로 옮길 예정.

## 6. 홈 첫 화면 신선도 트리거 (재방문 #3)

### 6-1. "지난 방문 이후 새 글 N개" 배지

`HomeAuthed.tsx`에 localStorage 기반 마지막 방문 시각 저장. 진입 시 그 이후 created_at 글 카운트. 인기 게시글 섹션 위에 그라데이션 칩으로 표시 → 클릭 시 `/community` 이동. 0개면 자동 숨김. 첫 방문 유저는 비교 기준 없으니 안 보임 (자연스러움).

### 6-2. 푸시 권한 권유 카드 (PushOptInCard.tsx)

기존 `PushSubscriber.tsx`는 로그인 즉시 `Notification.requestPermission()`을 자동 호출했음 — 신규 가입자가 첫 진입 직후 의미 없이 거부하고, 그 후 영구 차단되는 공격적 UX.

수정:
- `PushSubscriber` — 권한이 이미 `granted`인 사람만 자동 구독. default/denied는 건드리지 않음.
- 신규 `PushOptInCard` — `permission === "default"`일 때만 "내 글에 댓글 달리면 알려드릴까요?" 카드 노출. **사용자가 명시적으로 [켜기] 클릭**할 때만 권한 요청. dismiss 시 7일간 안 보임.
- 정확히 5번(댓글 푸시)이 의미 있게 동작하기 위한 follow-up.

## 7. SEO — "도시공존" 키워드 강화

배경: 사용자 요청 — "네이버에 길고양이 도시공존 지도 치면 우리 사이트가 나오는데 도시공존만 쳐도 나오게."

진단: WebSearch로 확인하니 "도시공존" 단독은 일반명사처럼 인식돼서 "도심공동화", 위키, 학술자료 등이 상위 점유. long-tail("도시공존 길고양이 지도")은 우리가 사실상 유일한 정답이라 무조건 1순위.

코드 측면 작업:
- `app/components/HomeLanding.tsx` — 히어로 brand badge에 `"도시공존 ·"` 명시. 설명 첫 문장이 `"**도시공존**은 ..."`로 시작.
- `app/layout.tsx` — JSON-LD Organization 강화: `alternateName: ["도시공존", "DosiGongzon", "도시 공존"]`, `brand`, `email`, `contactPoint`, `knowsLanguage`. WebSite에 `publisher`, `alternateName` 추가.

운영 액션 (사용자 차례):
1. 네이버 서치어드바이저 — `/`, `/about`, `/map`, `/areas` 수동 수집 요청. 사이트맵 등록.
2. **네이버 사이트 검색 등록 신청** ⭐⭐⭐ — 통과되면 단독 "도시공존" 검색 시 "사이트" 영역에 박힘. 가장 큰 한 방.
3. 네이버 블로그/카페 백링크 작성 (네이버는 자기네 콘텐츠 강력 우대).

기다리는 시간 보통 2~4주. 신생 도메인은 1단어 매칭 늦게 풀림. long-tail부터 조금씩 풀리고 결국 단독도 풀리는 순서.
