## 1. Sentry 에러 2건 핫픽스 (커밋 `599da32`)

아침에 메일 두 통.

### DOSIGONGZON-1: `Could not find the table 'public.push_alert_log'`
`cron/storage-diet/push_alert_log` scope에서 발생. `push_alert_log` 테이블이 Supabase 프로덕션에 마이그레이션 안 된 상태. SQL은 `box/supabase_push_alert_log_migration.sql`에 있었는데 실행이 누락됨. health-alert-push도 같은 테이블 쓰는데 그쪽은 try-catch + reportError 없어서 조용히 실패 중이었던 듯 — Sentry 신호도 없이 dedup이 작동 안 하던 위험 상태.

**조치:**
- `storage-diet/route.ts`에서 PGRST205 (schema cache miss) 코드는 조용히 skip — SQL의 `to_regclass` 가드와 동일 의도
- 사용자에게 SQL Editor 실행 안내 → 실행 완료 확인

### DOSIGONGZON-2: `SyntaxError: Unexpected token '<', "<!DOCTYPE..."`
`cron/sync-pharmacies`. LOCALDATA 공공 API가 200 OK + HTML(점검·에러 페이지)을 리턴해서 `res.json()`이 폭발.

**조치:** Content-Type이 `json` 포함 안 하면 graceful skip + 다음 실행 재시도.

## 운영 메모

- **RSS는 빠른 발견 도구**. Feedly·NetNewsWire 자동 발견되니 외부 자동화에서 도시공존 콘텐츠 활용 가능. 다만 Google News Sitemap은 별도. 등재 신청 후 처음 4주는 트래픽 미미하더라도 정상.
- **동적 OG ↔ metadata.openGraph.images 충돌** 함정 주의. 명시적 images가 있으면 같은 폴더 `opengraph-image.tsx`가 무시됨. 동적 OG 쓰려면 명시적 images를 endpoint로 가리키게.
- **ConsentManager 패턴**: 한국 법규는 "쿠키 사용 자체"에 동의받기 요구라 Vercel Analytics가 익명이어도 동의 후 로드해야 안전. 한 컴포넌트에서 배너 UI + Analytics 게이팅 다 처리하면 깔끔.
- **MarketingConsentApplier ↔ PendingInviteApplier 패턴** 재활용. OAuth redirect 거치는 동의·코드는 sessionStorage에 두고 user mount 시 처리하는 게 가장 안정적.
- 14세 미만 안내 mailto에 subject·body 프리필해두니 보호자가 작성 부담 적음. URL encode 정성껏.

## 10. 변경분 세밀 검토 + 버그 6건 일괄 가드 (커밋 `6b6edba`)

"버그 찾아보자 아주아주아주아주 세밀히" 요청. Explore subagent 3개로 병렬 진단(오늘 변경 회귀 / 보안·RLS·인증 / UX·일관성·빌드). 핵심 보안·SEO 인프라는 양호하다고 나왔으나, 어제 신규 추가한 코드의 엣지 케이스 6건이 잡혔다.

### 즉시 수정 4건

- **ConsentManager SSR/CSR hydration mismatch** — `mounted` state 추가해서 hydration 전엔 null 렌더. Analytics가 서버엔 없고 클라이언트에만 마운트되어 React 경고/에러 발생 가능했음.
- **MarketingConsentApplier 무한 retry** — 성공·실패 무관 sessionStorage 무조건 제거 + 실패는 Sentry로 보고. RLS 또는 네트워크 에러 시 매 세션 시도해서 에러 누적되던 구조 차단.
- **동적 OG 3종 (tips/news/shorts) null 크래시** — try/catch로 fetch 감싸 삭제된 글 공유 시 500 대신 fallback OG 렌더. `getTipBySlugServer`/`getNewsById`/`getPublishedShortServer` null 반환 또는 throw 모두 대응.
- **RSS 피드 Supabase 클라이언트 이중 생성** — 뉴스용·숏츠용으로 두 번 `createClient()` 호출하던 거 한 번으로 통합. connection pool 절약.

### 빠른 수정 2건

- **Signup 페이지 mount 시 sessionStorage stale 클리어** — 사용자 A가 가입 도중 cancel하고 사용자 B가 같은 탭에서 가입할 때 마케팅 동의 의도가 B에게 적용되는 미세한 가능성 차단.
- **News 카드 출처명 line-clamp-1** — 매우 긴 `source_name`에서 카드 깨짐 방지.

### 검토에서 OK로 확인된 영역

- 핵심 API 인증·RLS·CRON_SECRET·service_role 분리·URL 검증·CORS·OAuth state — 견고
- FAQ 페이지 내부 링크 13개 모두 실제 라우트 존재
- 냥숏츠→동물숏츠 리브랜딩 잔존 없음
- 사이트맵 /faq 추가 정상 prerender

## 11. 해킹·도배 방어 진단 + 핵심 5건 가드 (커밋 `cbb6acf`, `f7e65bb`)

"해킹방어시스템도 고려해봐야하지않겠어? 도배라던지" 요청. 다시 Explore subagent 3개로 (rate limit·spam / bot·upload·XSS / 계정 무결성·어뷰즈). 결과 진짜 위험 한 가지 발견.

### 발견 — `auto_hide_reported_target()` 트리거 악용 (가장 위험)

이미 적용돼 있던 자동 숨김 시스템: **신고 3건 누적 시 콘텐츠 자동 숨김**. 그런데 신고 중복 방지가 전혀 없었음. → **봇 3개 또는 알트 계정 3개로 정상 사용자 콘텐츠를 즉시 숨김 처리 가능**. 자동 숨김 시스템이 오히려 공격 무기가 되어있던 상태.

### 적용한 방어 5종 (`box/supabase_abuse_defense_migration.sql`)

PostgreSQL BEFORE INSERT trigger로 처리. 사용자 친화적 한국어 RAISE EXCEPTION 메시지.

1. **신고 도배 방지** — 같은 (reporter, target) 24h 1회 + 일일 10건
2. **정지 유저 신고 차단** — `reports_insert_auth` 정책에 `is_user_not_suspended(auth.uid())` 추가
3. **DM 도배** — 수신자당 1분 5건 + sender 시간 30건
4. **게시글 도배** — author 시간 5건
5. **댓글 도배** (post_comments + cat_comments) — author 시간 20건

### 함정 — `is_user_not_suspended()` 함수 시그니처

처음 작성 시 `is_user_not_suspended()` (인자 X)로 호출했는데, 실제 함수는 `is_user_not_suspended(uid uuid)` 였음. 사용자가 실행하다 `42883: function does not exist` 에러로 멈춤. 한 번 수정해서 `auth.uid()` 인자 명시 — `f7e65bb`. SQL이 idempotent라 다시 처음부터 실행 OK.

또 다른 함정: 사용자가 SQL Editor에 내가 채팅에 보여준 BEFORE/AFTER 설명 코드 블록까지 같이 붙여넣어서 `syntax error at or near "and"` 발생. 그 블록은 단순 설명이고 실행하면 안 된다고 안내 — 파일 통째로 다시 복사하라고 안내해서 해결.

### Cloudflare Turnstile signup 적용

별도로 발견된 큰 누락. CLAUDE.md에 "봇 방어: Cloudflare Turnstile (signup)"라고 적혀 있고 검증 API(`/api/turnstile-verify`)도 준비돼 있는데 **실제 signup 페이지에 위젯이 없었음**. 코드와 문서 불일치.

`TurnstileWidget` import + 약관 동의 다음에 렌더. `handleSignup` 시 토큰을 서버 검증 후 통과해야 OAuth 진행. `NEXT_PUBLIC_TURNSTILE_SITE_KEY` 미설정 환경(개발)에선 자체 bypass.

Vercel 환경변수 확인 결과: `NEXT_PUBLIC_TURNSTILE_SITE_KEY` + `TURNSTILE_SECRET_KEY` 둘 다 Production·Development에 31일 전부터 등록되어 있었음 → 봇 차단이 실제로 동작 시작.

## 운영 메모 (오후 추가분)

- **`auto_hide_reported_target()` 같은 자동 트리거는 신중히 도입해야 함**. 임계값(N건)이 있는 자동 조치는 도배 방지가 없으면 오히려 공격 도구가 됨. 도시공존이 5/20 오픈 전에 잡힌 게 천만다행.
- **PostgreSQL trigger의 RAISE EXCEPTION 메시지는 그대로 사용자에게 노출됨**. Supabase는 error.message 그대로 전달하니 한국어로 작성해두는 것이 UX 좋음.
- **CLAUDE.md ↔ 실제 코드 불일치 패턴** 주의. 문서에 "적용됨"이라 적혀 있어도 실제 컴포넌트 import·렌더 여부 확인이 필요. Turnstile은 검증 API만 있고 UI가 없는 상태로 31일 동안 방치되어 있었음.
- **함수 시그니처 확인 없이 SQL 작성 금지**. `is_user_not_suspended()` 처럼 인자 유무가 다르면 즉시 에러. 다른 마이그레이션 파일에서 호출 형태를 grep으로 한 번 확인하는 게 안전.
