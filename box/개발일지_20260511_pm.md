## 1. OAuth 로그인 실패 분석 + 진단 강화 (커밋 `01741be`)

### 진단
사용자가 보낸 `/admin/auth-errors` 캡처:
- Top 1: auth_failed (4건, Chrome Android, stage=redirected)
- Top 2: exchange_failed (4건, google, Chrome Android, stage=exchange)
- Top 3: access_denied (2건, google, Safari iOS, 1시간 전)

분석 결과:
- **#3 access_denied** = 사용자가 Google 동의 거절. 실 버그 X.
- **#1, #2는 사실상 같은 흐름** — exchange_failed가 발생하면 callback이 `/login?error=auth_failed`로 리다이렉트하고 login 페이지가 또 한 번 stage=redirected로 기록. 즉 실 케이스는 4건.
- "브라우저 저장소가 차단됐어요" 메시지가 떴다는 건 Supabase가 반환한 desc에 "cookie/storage" 키워드 포함 = **PKCE code verifier 쿠키가 callback 시점에 없음**.

### 진단 강화 — callback/route.ts
다음 발생 시 원인 즉시 판별하도록 `error_desc` suffix에 자동 추가:
```ts
const diag = await getDiagSuffix();
// [diag:verifier=yes|no;cookies=N]
const fullDesc = `${exchangeError.message}${errCode ? ` (code:${errCode})` : ""}${errStatus ? ` (status:${errStatus})` : ""}${diag}`;
```

이로써:
- `verifier=no` → 쿠키 손실 (시크릿 모드, TWA→Custom Tabs 컨텍스트 전환, 사용자가 OAuth 중 다른 탭/앱 전환)
- `verifier=yes` + Supabase code/status → 진짜 Supabase 검증 실패 (code 만료/재사용/키 불일치)

추가로 code 파라미터 없이 callback 진입한 비정상 케이스도 신규 stage `callback_no_code`로 분리 기록.

### 재시도 UX — login/page.tsx
에러 배너 안에 ↻ "다시 시도" 버튼 추가. 에러 쿼리만 제거하고 `next`는 보존해 깨끗한 상태로 복귀. access_denied(사용자 취소)·auth_failed·exchange_failed 모두에 적용.

## 6. 로그인 next 파라미터 5곳 추가 (커밋 `7c80884`)

### 마찰
비로그인 유저가 어떤 기능을 시도하다 로그인하면 홈(`/`)으로 가는 곳이 6개 발견:
1. `AddCatModal` 비로그인 안내 (지도에서 + 누른 비로그인)
2. `/map` 비로그인 안내 배너
3. `/mypage` 비로그인 초대 카드
4. `/mypage/inquiries` 비로그인 안내
5. `/mypage/blocked-users` 비로그인 안내
6. `PublicHeader` (헤더에서 로그인 클릭)

### 수정 (5건)
각각 적절한 `next` 파라미터 추가:
- AddCatModal: `/login?next=%2Fmap`
- map 배너: `/login?next=%2Fmap`
- mypage 카드: `/login?next=%2Fmypage`
- mypage/inquiries: `/login?next=%2Fmypage%2Finquiries`
- mypage/blocked-users: `/login?next=%2Fmypage%2Fblocked-users`

callback에서 `safeNextPath`로 이미 검증되므로 보안 OK.

### 미처리
`PublicHeader` (서버 컴포넌트, 모든 공개 페이지에서 사용)는 현재 pathname을 동적으로 받아야 next 가능. 별도 작업으로 분리 — `headers()`로 referer 활용 또는 props 추가 등 결정 필요.
