## 1. 프로젝트 비전 & 전략

### 1.1 한 줄 요약

> 길고양이는 지자체 동물 민원의 1위 원인이지만, 현장의 해결 주체인 시민 케어러에게는 아무런 시스템이 없습니다. **'도시공존'은 케어러에게 기술을 쥐여주어, 민원을 민원으로 끝내지 않고 공존으로 바꾸는 플랫폼입니다.**

### 1.2 비전

기술(하드웨어 스마트 쉘터)과 공동체(소프트웨어 플랫폼)가 결합된, 지속 가능한 도시 길고양이 돌봄 생태계를 구축한다. 지자체의 동물 관련 민원 부담을 줄이고, 시민 케어러의 자발적 돌봄을 체계화하여 길 위의 생명과 인간이 안전하게 동행하는 도시를 만든다.

### 1.3 핵심 전략: 하드웨어-플랫폼 플라이휠

- **Software (플랫폼):** 정보 공유·커뮤니티·지역 소모임을 통해 시민 케어러를 결집하고, 잠재적 쉘터 수요자(개인/지자체)를 확보한다.
- **Hardware (스마트 쉘터):** 앱 없이는 관리 불가능한 IoT 쉘터(발열·CCTV·감지·알림)를 통해 유저를 플랫폼에 강력하게 락인한다.
- **플라이휠:** 커뮤니티 결집 → 케어러 수요 가시화 → 쉘터 렌탈 증가 → 쉘터 데이터가 플랫폼 유입 → 플랫폼 가치 상승 → 신규 유저 유입 → (반복).

### 1.4 CCTV 사용자 모니터링 체제 (장기 비전)

스마트 쉘터의 CCTV를 운영자(1인)가 감시하는 것이 아니라, **해당 구역 케어러들이 직접 모니터링**하는 구조로 설계한다. "우리 동네 고양이 잘 있나" 확인하는 행위가 자연스러운 이상 징후(고장, 학대 위협) 제보로 이어지며, 운영 인력 없이 감시 네트워크가 형성된다. 단, CCTV 연동은 하드웨어 출시 이후 단계에서 개발한다.

### 1.5 해결하고자 하는 핵심 문제

| 문제 | 현황 | 도시공존의 해결 방식 |
|------|------|---------------------|
| **민원의 악순환** | 길고양이 관련 민원은 지자체 동물 민원의 최다 비중을 차지하지만, 지자체는 현장 대응 인력이 부족하고, 시민 케어러는 체계적 지원 없이 개인 역량에 의존. | 케어러에게 구조 가이드·커뮤니티·위치 보안을 제공하여 현장 대응력을 높이고, 지자체에는 "시민이 직접 해결하는 모델"을 제안. |
| **정보의 파편화** | 위급 상황 시 신뢰할 수 있는 가이드를 찾는 데 골든타임을 허비. | 보호지침 도움센터에서 즉각적인 응급·포획·법률 가이드 제공. |
| **심리적/경제적 고립** | 구조·치료 비용을 개인이 전부 부담하며 발생하는 번아웃. | SOS 펀딩, 지역 소모임을 통한 비용·책임 분산. |
| **사회적 갈등** | 비체계적 돌봄으로 인한 지역 주민 마찰 및 법적 지식 부재. | 급식소 관리 수칙, 법률 가이드를 법적 근거와 함께 제공. |
| **돌봄 인프라 부재** | 겨울철 동사, 차량 사고 등 물리적 위험에 대한 보호 수단 없음. | 스마트 쉘터(발열·감지·알림)로 물리적 돌봄 환경 혁신. |

## 2. 팀 구성 및 리소스

### 2.1 개발 체제

- **1인 풀스택 개발**, AI 코딩 어시스턴트 활용.
- 디자인: 본인 직접 수행 (벤토 스타일 레퍼런스 기반).
- 수의사 자문: 외부 1~2명 (AI 수의사 응답 템플릿 검수, 유상 자문).
- 법률 검토: 외부 법률 자문 1회 (이용약관·면책 조항 검수).

### 2.2 1인 개발 원칙

- **MVP는 최소한으로 자른다.** 한 스프린트(2주)에 1개 기능만 집중.
- **완성도보다 출시 속도를 우선한다.** 디자인 80%에서 런칭, 유저 피드백으로 개선.
- **매니지드 서비스를 최대한 활용한다.** Supabase Auth/Realtime/Storage, Vercel 배포.
- **CCTV·IoT 등 하드웨어 연동 기능은 하드웨어 출시 전까지 개발하지 않는다.**

## 3. 비즈니스 및 수익 모델

### 3.1 단계별 수익 전략

| 단계 | 수익원 | 설명 |
|------|--------|------|
| **Phase 1 (현시점~런칭)** | 정부 초기 창업 지원금 | 개발·운영비의 주 재원. 월 운영비 5만원 미만 구조. |
| **Phase 2 (하반기)** | 크라우드펀딩 + 지자체 Pilot | 스마트 쉘터 양산 자금 확보 + B2G 시범 1~2건. |
| **Phase 3 (정식 서비스)** | **쉘터 렌탈 + SaaS 월 구독 (핵심)** | 기기 렌탈료 + 플랫폼 데이터 관리비를 결합한 월 과금. |
| **Phase 3+** | 부가 수익 | SOS 펀딩 수수료(3~5%), 동물병원 제휴 광고, 지자체 데이터 컨설팅(B2G). |

### 3.2 SaaS 과금 모델 (상세)

| 구분 | 내용 |
|------|------|
| **과금 구조** | 쉘터 기기 렌탈료(월) + 플랫폼 구독료(월)를 단일 청구서로 통합. 유저 입장에서는 "쉘터 월 이용료"로 인식. |
| **B2C (개인 케어러)** | 기기 렌탈 + 앱 모니터링(CCTV·센서·알림) 포함 월 정액 (가격은 원가 확정 후 설정). |
| **B2B (아파트·캠퍼스)** | 복수 쉘터 관리 + 관리사무소 대시보드 제공. 볼륨 디스카운트. |
| **B2G (지자체)** | 연간 서비스 계약 형태. '동물복지 증진' 또는 '유기동물 관리' 세부사업비에 편입. 기기 납품(물품) + 플랫폼 유지보수·데이터 관리(용역)로 분리하여 조달 절차에 대응. |

## 4. 경쟁 및 대안 분석

### 4.1 현재 케어러들이 사용하는 도구

| 대안 | 장점 | 한계 |
|------|------|------|
| 네이버 카페 (길냥이를 부탁해 등) | 대규모 회원, 축적된 정보 | 검색 어려움, 지역 기반 협력 불가, 긴급 대응 없음 |
| 카카오톡 단톡방 | 실시간 소통 | 정보 휘발, 신규 진입장벽, 체계적 기록 불가 |
| 인스타그램/블로그 | 개인 기록·공유 | 협력 기능 없음, 커뮤니티 구조 아님 |

### 4.2 도시공존의 차별화

- **정보 → 행동 → 협력**이 하나의 플로우로 연결되는 유일한 플랫폼.
- 스마트 쉘터와 연동되는 **하드웨어-소프트웨어 통합 생태계** (직접 경쟁자 부재).
- AI 기반 즉각 가이드 (기존 카페/단톡방의 "답변 대기" 해소).
- **지역별 소모임**으로 같은 동네 케어러 연결 (카톡 단톡방의 폐쇄성 해결).
- 위치 보안 시스템으로 학대 방지까지 고려 (기존 대안에 없는 기능).
- **지자체 민원 해결이라는 공공적 명분**을 갖춘 B2G 진입 가능 모델 (기존 커뮤니티에는 없는 사업 구조).

## 7. 핵심 사용자 여정 (Critical User Journeys)

### 7.1 위급 상황 구조 시나리오

1. **발견:** 시민이 다친 고양이를 발견하고 [보호지침 도움센터]에 접속.
2. **진단:** '응급 가이드' 확인 후, AI 수의사에게 현재 상태(사진/설명)를 물어 즉각 조치법 숙지.
3. **제보:** [커뮤니티 > 긴급 제보]에 정확한 위치와 사진을 게시.
4. **협력:** 같은 지역 소모임 수호대원에게 알림 발송. 상단 [공지: 포획 가이드] 확인, 포획틀 지원 요청.
5. **완료:** 구조 성공 후 병원 이송 및 경과를 공유하며 공동체의 응원을 받음.

### 7.2 일상 돌봄 시나리오

1. **기록:** 케어러가 매일 밥자리를 방문하고, 사진 1장 + 한 줄 메모로 돌봄 일지 작성.
2. **공유:** 같은 구역 소모임 멤버들이 피드에서 확인. "오늘 3번 밥자리 비었네" → 다른 멤버가 대신 방문.
3. **축적:** 시간이 지나면 특정 고양이의 건강 시계열 데이터가 쌓임. 수의사 방문 시 보여줄 수 있는 실용적 기록.

### 7.3 유저 스토리

- "활동가인 나는 아이들을 안전하게 구조하기 위해, 포획틀 사용법을 단계별로 정확히 알고 싶다."
- "초보 돌봄이인 나는 이웃과의 마찰을 피하기 위해, 급식소 관리 수칙을 법적 근거와 함께 확인하고 싶다."
- "아파트 관리소장인 나는 주민 민원을 줄이기 위해, 스마트 쉘터를 설치하고 앱으로 관리 현황을 확인하고 싶다."
- "같은 동네 케어러인 나는 혼자 돌보는 불안감을 줄이기 위해, 지역 소모임에서 서로의 돌봄 현황을 확인하고 싶다."

### 8.2 커뮤니티 카테고리 (9개)

1. **긴급 제보** (구조 요청, 사진 첨부 필수)
2. **TNR 정보 공유**
3. **급식소/쉼터 관리**
4. **임시보호/입양**
5. **동물병원 후기·추천**
6. **법률/행정 Q&A**
7. **자유게시판**
8. **성공 스토리** (구조 후기)
9. **지역별 소모임**

### 8.3 지역별 소모임 (상세)

**목적:** 같은 동네 케어러들을 연결하여, 카톡 단톡방의 폐쇄성과 정보 휘발 문제를 해결.

**설계:**

- **자동 매칭:** 가입 시 활동 지역(구/동 단위)을 선택하면, 해당 지역 소모임에 자동 배정.
- **소모임 피드:** 해당 지역의 긴급 제보, 돌봄 일지, 구역 공유 현황이 모아서 노출.
- **소모임 공지:** 지역 리더(자발적 신청 + 활동 이력 기반 승인)가 동네 TNR 일정, 쉘터 위치 안내 등 작성.
- **핵심 가치:** "내가 못 가는 날 같은 동네 누군가가 밥자리를 확인해줄 수 있다"는 안전망.

**MVP 범위:** 소모임 카테고리 + 지역 필터링까지만 구현. 지역 리더 시스템은 유저 풀 확보 후 도입.

### 8.4 커뮤니티 운영 정책 (모더레이션)

- 신고 누적 3회 시 게시물 자동 블라인드, 관리자 검토 후 처리.
- 욕설·혐오·학대 조장 콘텐츠는 즉시 삭제 및 계정 정지.
- TNR 찬반, 급식소 갈등 등 민감 주제 토론 가이드라인 사전 게시.
- 긴급 제보 카테고리는 허위 제보 방지를 위해 사진 첨부 필수.
- **1인 운영 대응:** 초기에는 유저 신고 기반 사후 관리. 소모임별 지역 리더가 확보되면 1차 모더레이션 위임.

## 9. 특화 설계: 리텐션 & 보안 전략

### 9.1 데일리 돌봄 일지 (Daily Care Log)

구조라는 비정기적 이벤트 외에 유저가 매일 들어올 이유를 제공한다.

**핵심 원칙: 기록이 나만을 위한 것이 아니라 공동체를 위한 것이어야 동기가 유지된다.**

- **구역 공유:** 내가 돌보는 구역을 등록하면, 같은 지역 소모임 내 다른 수호대원과 연결.
- **간편 기록:** 사진 1장 + 한 줄 메모로 완료. 복잡한 양식 없음.
- **AI 보조 (자동 생성 아님):** 사진 업로드 시 AI가 키워드·요약을 제안하지만, 유저가 직접 수정·확인 후 게시.
- **건강 시계열:** 돌보는 고양이의 건강 변화를 시간순 기록. 수의사 방문 시 보여줄 수 있는 실용적 가치.
- **리워드 (Phase 3 이후만):** 포인트는 쉘터 출시 후 렌탈료 할인·제휴 병원 쿠폰으로 교환. 쉘터 출시 전에는 포인트 시스템을 도입하지 않는다.

### 9.2 점진적 보안 인증 (Progressive Auth)

| 레벨 | 조건 | 권한 |
|------|------|------|
| **Lv.1 게스트** | 소셜 로그인 | 가이드·게시판 조회. 제보 위치는 100m 반경 흐림. |
| **Lv.2 인증 대원** | 휴대폰 본인 인증 | 제보 글 작성, 상세 위치(핀) 확인, 돌봄 일지 작성. |
| **Lv.3 쉘터 관리자** | 활동 이력 검증 + 쉘터 렌탈/구역 등록 | 해당 쉘터 CCTV·센서 접근 (Phase 3 이후). |

**MVP:** Lv.1과 Lv.2만 구현. Lv.3는 하드웨어 출시 후.

### 9.3 긴급 모드 (Lock-down)

학대 위협 감지 시, 관리자가 특정 지역의 제보 글을 즉시 비공개 처리. 해당 지역 소모임에 경고 공지 발송.

## 10. AI 수의사 리스크 관리

### 10.1 서비스 범위

이 기능은 "AI 진단"이 아니라 **"응급 행동 가이드"**이다.

| 허용 | 차단 |
|------|------|
| "이런 증상이면 이렇게 응급 처치하세요" | 특정 질병명 확정 진단 |
| "가장 가까운 24시 동물병원은 여기입니다" | 약물·용량 관련 답변 |
| "이 상태는 긴급합니다, 즉시 병원으로" | "치료하지 않아도 됩니다" 류의 안심 답변 |

### 10.2 대응 구조

- **빈번한 응급 상황:** 수의사 검수를 거친 사전 작성 가이드를 즉시 제공. AI 생성이 아닌 검증된 콘텐츠.
- **복잡한 사례:** AI 분석 후 반드시 "가까운 동물병원 방문" 권고.
- **강제 면책 고지:** 모든 채팅 시작/끝에 "의료적 진단이 아니며 법적 책임을 지지 않습니다" 노출 + 24시 병원 버튼 상시 배치.
- **품질 모니터링:** "위험/잘못된 정보" 유저 신고 별도 트래킹. 위험 안내 오류율 0% 목표.

## 12. B2G 시범 사업 진입 전략

### 12.1 타겟 및 명분

- **1차 타겟:** 서울시 자치구 중 길고양이 민원이 많고 TNR 예산이 편성된 구 (예: 마포구, 성동구, 강동구).
- **편입 가능 예산:** 각 지자체 '동물복지 증진' 또는 '유기동물 관리' 세부사업비.
- **핵심 명분:** "지자체 동물 민원 1위인 길고양이 문제를, 시민 케어러가 앱으로 직접 해결하게 해주는 모델." 지자체 담당자의 업무 부담 경감이 핵심 셀링 포인트.

### 12.2 진입 타임라인

| 시점 | 액션 |
|------|------|
| 5~6월 (MVP 런칭 후) | 플랫폼 DAU·커뮤니티 활성도 데이터를 기반으로 제안서 초안. |
| 7월 | 타겟 지자체 담당 부서 미팅 및 Pilot 제안. |
| 8월 | 크라우드펀딩과 동시에 시범 설치 1~2기 무상 제공 (지자체 비용 부담 최소화). |
| 9~10월 | Pilot 운영 데이터 수집 (쉘터 이용 빈도, 민원 감소 효과 측정). |
| 11월~ | Pilot 결과를 근거로 본계약 또는 차년도 예산 편성 제안. |

> **진입 원칙:** "사달라"가 아니라 "시범 1~2대 무상 설치, 데이터를 같이 보자"로 접근. 초기 비용은 크라우드펀딩 자금에서 감당.

## 13. 기술 설계 원칙: 실용적 클린 아키텍처

1인 개발의 속도를 사수하기 위해 **'선택적 분리'** 전략을 취한다.

### 13.1 계층별 설계

| 계층 | 원칙 | 설명 |
|------|------|------|
| **Domain (엄격 분리)** | 외부 기술 의존 없음 | `Cat`, `Shelter`, `User` 엔티티 + 핵심 비즈니스 로직(위치 권한 검증, 보안 등급 판정). 앱 개발 시 100% 재사용. |
| **Presentation (유연)** | Next.js App Router | 단순 조회는 서버 컴포넌트. 복잡한 흐름만 유즈케이스 경유. |
| **Infrastructure (자동화)** | 매니지드 서비스 | Supabase Auth·Realtime·Storage, Vercel CI/CD. |

### 13.2 기술 스택 선정 이유

| 기술 | 선정 이유 |
|------|-----------|
| **Next.js** | SSR/SEO → '냥줍', '길고양이 구조' 검색 유입 극대화. 프론트·백엔드 통합 관리. |
| **Supabase** | Auth·Realtime·Storage 통합. Free tier 초기 비용 0원. Realtime 긴급 제보 푸시. |
| **Vercel** | Next.js 네이티브 배포. CI/CD 자동화. |
| **React Native (추후)** | Domain Layer 재사용으로 앱 개발 가속화. |

### 13.3 개인정보 및 법적 사항

- 제보 시 상세 주소 노출 기본값은 비공개 (Lv.2 이상만 핀 확인).
- 보호법률 가이드는 최신 농림부 지침 기반 업데이트.
- AI 수의사 이용약관 면책 조항 (법률 자문 1회 반영).
- CCTV 영상 관련 개인정보 처리방침은 하드웨어 출시 전 별도 수립.
